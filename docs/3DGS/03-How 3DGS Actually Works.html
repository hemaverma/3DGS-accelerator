Overview

The Four-Step Engine

Everything in 3DGS is one repeating loop. Four steps, running thousands of times until the scene looks right.

The Core Loop — Repeated ~30,000 times during training

Each iteration picks a training photo, renders it from Gaussians, compares to ground truth, then adjusts every parameter via gradients

Step 1 · Projection

Taking a 3D Ellipsoid into 2D

The first mechanical step: for each Gaussian, project its 3D ellipsoid shape onto the image plane. The 3D covariance becomes a 2D covariance — a smaller, simpler ellipse on screen.

Think of it like shining a flashlight at an egg. The shadow on the wall is always an ellipse, just flatter. The math that does this is called the EWA splatting approximation — it's fast and keeps the projection differentiable.

3D Ellipsoid → Camera Space → 2D Screen Ellipse

The projection changes shape depending on the viewing angle — a sphere stays round, but a disk tilted away from the camera gets squashed into a thin sliver

Why Viewing Angle Changes the 2D Shape

The same flat-disk Gaussian looks different at 0°, 45°, and 90° — this view-dependence is what makes the representation so powerful

Σ' = J · W · Σ · Wᵀ · Jᵀ
Σ' = 2D projected covariance (the screen ellipse shape)  ·  Σ = 3D covariance (the blob shape)  ·  W = camera rotation  ·  J = Jacobian of perspective projection (handles the "squish")

Intuition: We're asking "if I look at this 3D ellipsoid through this camera, what ellipse do I see?"
Step 1b · Under the Hood

How Shape Is Stored: R · S · Sᵀ · Rᵀ

Here's a subtle but important point. We can't just store the covariance matrix Σ directly and optimize it — we'd risk getting a physically impossible (non-positive-definite) matrix. Instead, 3DGS decomposes it into two things you can safely optimize:

Scale S — 3 numbers (sx, sy, sz) Rotation R — 4 numbers (quaternion)

Then reconstruct: Σ = R · S · Sᵀ · Rᵀ. This is always valid. You can push any numbers through scale and rotation and the result is always a legal covariance. This is why training is stable.

Scale × Rotation = Final Ellipsoid Shape

Scale stretches the sphere into an ellipsoid. Rotation tilts it. The final shape is the composition of both — always a valid ellipsoid.

Step 2 · Sorting

Depth Sorting — Getting the Order Right

Before blending, we must draw back-to-front. The farthest Gaussian gets painted first, then closer ones on top. This is called painter's algorithm. Get it backwards and the colors blend wrong — you'd see background colors bleeding through foreground objects.

3DGS sorts all Gaussians by their distance from the camera every frame. With millions of Gaussians, this is a serious GPU radix sort — one of the most expensive steps.

Correct vs Incorrect Depth Order

Back-to-front (correct) vs front-to-back (wrong) — the compositing math only works in the right order

Step 3 · Blending

Alpha Compositing — The Blending Formula

Once sorted, each 2D Gaussian splat contributes color to each pixel it covers. The contribution depends on two things: how much the Gaussian's 2D Gaussian function covers that pixel (its weight at that position), and its opacity α.

C = Σᵢ cᵢ · αᵢ · Tᵢ
C = final pixel color  ·  cᵢ = color of Gaussian i  ·  αᵢ = opacity × 2D Gaussian weight at this pixel  ·  Tᵢ = transmittance = how much light is left (1 − Σ previous α's)

Transmittance is the key insight: each Gaussian only gets to contribute as much as the light that wasn't already blocked by everything in front of it. The first Gaussian gets full light. The second gets whatever's left. And so on — it trails off naturally.

How Transmittance Decays Through Layers

T starts at 1.0 and decays as each layer absorbs light — deeper Gaussians have diminishing influence on the final pixel color

Step 4 · Learning

The Loss Function — How It Knows It's Wrong

After rendering, 3DGS compares the result to the ground truth training photo. The difference is the loss. Gradients from this loss flow backwards through every operation — all the way back to each Gaussian's 59 parameters.

L = (1-λ) · L₁ + λ · L_SSIM
L₁ = mean absolute pixel error (pushes colors toward truth)  ·  L_SSIM = structural similarity loss (preserves edges and texture)  ·  λ = 0.2 (SSIM weight) — default setting

Using both losses: L₁ alone can get the right average color but miss sharp detail. SSIM catches structure.
Rendered vs Ground Truth — What the Loss Sees

The error map (right) drives gradients — bright = big error, dark = well-converged. This is what nudges each Gaussian toward its final position.

What Gradients Flow Back Through

Every parameter of every Gaussian receives a gradient signal — telling it which direction to move to reduce error

The Special Sauce

Adaptive Density Control — Growing the Scene

Standard gradient descent only moves existing Gaussians. But 3DGS also adds and removes Gaussians every ~100 iterations based on the gradient signal. This is what grows a sparse point cloud into a dense photorealistic scene.

1
Detect under-reconstruction: Find Gaussians with large position gradients — the optimizer is straining to move them but can't get there with one blob.
2
Small Gaussian? → Clone it. Duplicate the Gaussian and offset the copy toward the gradient direction. You now have two small blobs where one was insufficient.
3
Large Gaussian? → Split it. Replace a big Gaussian with two smaller ones. Sample their positions from the original Gaussian's distribution, divide scale by 1.6.
4
Prune floaters: Every 3000 steps, reset opacity of all Gaussians — forcing each one to re-earn its existence. Near-zero opacity ones get deleted.
