Section 01

High-Level Overview

PM Lead Dev

The 3DGS Video Processor is a long-running Rust service that watches a designated input directory for new multi-video scene folders. When a complete upload is detected, it automatically extracts frames, computes camera poses via structure-from-motion (COLMAP), trains a 3D Gaussian Splatting model, and exports the result in PLY and SPLAT formats.

The system is designed for unattended, containerized operation with Azure Blob Storage integration, automatic retries, checkpoint-based resumption, and graceful shutdown.

Section 03

Stage 0 — Initialization & Configuration

PM Lead Dev
Stage 0 · Bootstrap

System Startup Sequence

The service starts by initializing structured logging, loading configuration from environment variables (with optional YAML override for training hyperparameters), optionally mounting Azure Blob Storage containers via Blobfuse2, and validating that all required directories exist.

  1. Initialize Logging — Sets up the tracing subscriber with structured JSON output. Log level is controlled by LOG_LEVEL env var (default: info). Credential values are automatically redacted.
  2. Load Configuration — Reads all operational parameters from environment variables (INPUT_PATH, OUTPUT_PATH, BACKEND, etc.). A separate YAML config file provides only 3DGS training hyperparameters (iterations, densification settings).
  3. Azure Blob Storage Mounting (optional) — If Azure config is detected, authenticates via connection string, SAS token, or Managed Identity, then uses Blobfuse2 to FUSE-mount blob containers as local directories.
  4. Validate Paths — Confirms that INPUT_PATH, OUTPUT_PATH, PROCESSED_PATH, ERROR_PATH, and TEMP_PATH all exist and are accessible.
  5. Start Health Server — Optionally launches an HTTP server on port 8080 with a GET /health endpoint returning JSON status (enabled via HEALTH_CHECK_ENABLED=true).
  6. Initialize Subsystems — Creates the shutdown coordinator, duplicate detector, job queue (capacity 100), and starts the retention cleanup scheduler.
  7. Spawn Event Loops — Launches the Watcher Loop and Processor Loop as concurrent Tokio tasks, then blocks on the main thread awaiting a shutdown signal.
Inputs
  • Environment variables
  • config.yaml (training params)
  • Azure credentials (optional)
Outputs
  • Configured runtime
  • Mounted storage (if Azure)
  • Health endpoint active
  • Watcher + Processor running
Potential Errors
  • Missing required env vars (INPUT_PATH, OUTPUT_PATH, etc.) — fatal, process exits
  • Azure authentication failure — fatal if Azure is configured
  • Blobfuse2 mount failure — fatal if Azure is configured
  • Path validation failure — fatal, directories must exist
  • Port 8080 already in use — health server fails to bind

Section 04

Stage 1 — File Watching & Folder Detection

PM Lead Dev
Stage 1 · Watcher Loop

Detecting New Scene Uploads

The watcher loop continuously monitors INPUT_PATH for new sub-folders. It uses inotify (Linux kernel filesystem notifications) for near-instant detection, with an automatic fallback to polling for network-mounted filesystems (NFS, Blobfuse2) where inotify may not work.

  1. Detect New Folder — Scans INPUT_PATH for directories that haven't been seen before. Uses inotify for local filesystems; falls back to polling at configurable intervals (POLL_INTERVAL_SECS, default 10s).
  2. Wait for Upload Stability — Once a folder is detected, the system monitors it for UPLOAD_STABILITY_TIMEOUT_SECS (default 60s) with no new file modifications. This ensures multi-video uploads from Azure Blob Storage or slow network connections are fully complete before processing begins. The timer resets each time a new file arrives.
Inputs
  • INPUT_PATH directory
  • Poll interval (10s default)
  • Stability timeout (60s default)
Outputs
  • Stable folder path ready for processing
Stability Detection Algorithm
stateDiagram-v2
  [*] --> Scanning: Start watching INPUT_PATH
  Scanning --> FolderDetected: New folder found
  Scanning --> Scanning: No new folders (poll interval)

Potential Errors
  • INPUT_PATH becomes inaccessible (Azure unmount, permission change)
  • Stability timeout never reached (continuously changing files)
  • inotify watch limit exceeded (falls back to polling automatically)

Section 05

Stage 2 — Duplicate Detection & Job Queuing

PM Lead Dev
Stage 2 · Dedup & Queue

Preventing Reprocessing & Managing Work

Before enqueuing a job, the system checks whether the folder has already been successfully processed or previously failed. This makes the service restart-resilient — no in-memory state is required for deduplication, as it checks the filesystem directly.

  1. Check processed/ — If a folder with the same name exists in PROCESSED_PATH, the job is skipped with status AlreadyProcessed.
  2. Check error/ — If a folder with the same name exists in ERROR_PATH, the job is skipped with status AlreadyFailed.
  3. Enqueue as New — A QueuedJob is created with a unique UUID and submitted to the bounded FIFO job queue (capacity: 100). If the queue is full, the enqueue call applies backpressure (waits).
Inputs
  • Detected folder path
  • PROCESSED_PATH directory
  • ERROR_PATH directory
Outputs
  • QueuedJob (UUID + folder path)
  • or: skip (already processed/failed)
Design Decision

Deduplication is filesystem-based, not in-memory. This means the service can be restarted at any time without losing track of what has been processed. Container restarts are safe.

Section 06

Stage 3 — Job Execution & Retry Logic

Lead Dev Core Dev
Stage 3 · Processor Loop

Orchestrating the Processing Pipeline

The processor loop dequeues jobs one at a time and runs the 8-stage pipeline within a retry wrapper. Before execution, disk space is verified. Each pipeline run supports checkpoint-based resumption — if a job is interrupted, it can resume from the last completed stage.

  1. Dequeue Job — Pulls the next job from the FIFO queue with a 1-second timeout (allowing periodic shutdown-flag checks).
  2. Check Disk Space — Verifies ≥ 10 GB free on the temp volume. If insufficient, the job is immediately moved to error/.
  3. Execute with Retry — Runs the full 8-stage pipeline. On retryable failure, retries with exponential backoff (2s → 4s → 8s → … capped at 60s), up to MAX_RETRIES (default 3). Non-retryable errors (validation, config) fail immediately.
  4. Move Result — On success, the input folder is moved to processed/. On final failure, it is moved to error/.

Stage 4 — Input Validation

PM Lead Dev
Pipeline Stage ① · Validation

Verifying Video Inputs

The first pipeline stage validates all video files in the input folder before any expensive processing begins. This is a non-retryable stage — validation failures indicate fundamentally bad input that will not improve on retry.

  1. Discover Videos — Scans the input folder for files with supported extensions (MP4, MOV, AVI, MKV, WebM).
  2. Validate Each Video — Checks format integrity, minimum resolution, minimum frame count, and minimum duration against configurable thresholds (ValidationConfig).
  3. Return Video Metadata — Extracts and returns metadata (resolution, FPS, duration, codec) for each valid video.
Inputs
  • Folder with video files
  • ValidationConfig thresholds
Outputs
  • List of validated VideoInput objects
  • VideoMetadata per video
Potential Errors (Non-Retryable)
  • No video files found in folder
  • Unsupported video format/codec
  • Resolution below minimum threshold
  • Too few frames or too short duration
  • Corrupted video file (FFprobe cannot read)

Stage 5 — Frame Extraction (FFmpeg)

PM Lead Dev Core Dev
Pipeline Stage ② + ③ · Frame & Metadata Extraction

Extracting Frames from Multi-Video Input

All videos in the input folder are processed concurrently. Each video is decoded by FFmpeg into a sequence of JPEG frames at a configurable frame rate. Metadata is extracted via FFprobe. Since FFmpeg is a blocking external process, all calls are wrapped in tokio::task::spawn_blocking to avoid starving the async runtime.

  1. Concurrent Extraction — All videos are processed simultaneously using futures::stream::buffer_unordered() with concurrency bounded by CPU count.
  2. FFmpeg Decode — For each video, runs:
    ffmpeg -i video.mp4 -vf "fps=N" -frames:v M -q:v 2 -an -y frame_%06d.jpg
  3. Metadata Extraction — FFprobe queries each video for resolution, duration, FPS, frame count, GPS coordinates, and camera information.
  4. Manifest Generation — Creates manifest.json with video entries, frame lists, and camera intrinsics. Supports user-provided camera_intrinsics.yaml override.
Inputs
  • N video files (MP4, MOV, etc.)
  • ExtractionOptions (FPS, max frames)
  • camera_intrinsics.yaml (optional)
Outputs
  • FrameSet per video (JPEG sequences)
  • VideoMetadata per video
  • manifest.json

Stage 6 — COLMAP Sparse Reconstruction

PM Lead Dev Core Dev
Pipeline Stage ⑤ · Structure from Motion

Computing Camera Poses & Sparse 3D Points

COLMAP is an industry-standard Structure-from-Motion (SfM) tool. This stage takes the extracted frames and computes the 3D position and orientation (pose) of each camera, along with a sparse point cloud. This geometric information is essential input for 3DGS training.

COLMAP runs as an external binary via spawn_blocking. The process is computationally intensive (minutes to hours depending on frame count and scene complexity).

  1. Feature Extraction — Detects visual keypoints (SIFT features) in every frame image.
    Command: colmap feature_extractor --image_path ... --database_path ...
  2. Feature Matching — Matches features between image pairs. Supports three strategies: exhaustive (all pairs, most accurate), sequential (adjacent frames, fastest), or vocab_tree (vocabulary-based, balanced).
  3. Sparse Reconstruction (Mapping) — Solves for camera poses and 3D point positions using bundle adjustment. Produces a sparse 3D model.
    Command: colmap mapper --database_path ... --image_path ... --output_path ...
  4. Validation — Verifies the sparse model has sufficient quality: minimum 1,000 3D points reconstructed. If insufficient, the error is non-retryable (indicates the input images don't have enough overlap or texture).
Inputs
  • Extracted JPEG frames
  • COLMAP binary path
  • Matching strategy config
Outputs
  • ColmapOutput (sparse model dir)
  • Camera poses (intrinsics + extrinsics)
  • Sparse 3D point cloud
  • ColmapStats (point/camera count)

Stage 7 — 3D Gaussian Splatting Training

PM Lead Dev Core Dev
Pipeline Stage ⑥ · 3DGS Model Training

Learning the 3D Gaussian Representation

This is the core computational stage. A pluggable backend takes the COLMAP sparse reconstruction and the extracted frames, then trains a 3D Gaussian Splatting model. The model represents the scene as millions of 3D Gaussians, each with position, covariance, color (spherical harmonics), and opacity. Training runs on GPU and is configured via YAML hyperparameters.

  1. Backend Selection — The system selects the training backend based on the BACKEND environment variable. If not specified, it auto-detects the GPU platform (CUDA, HIP, Metal) and selects the best available backend.
  2. GPU Detection — Queries for available GPU hardware and validates compatibility with the selected backend.
  3. Training Execution — Runs the backend's train() method with:
    • Frame images as training data
    • COLMAP camera poses as ground truth
    • YAML-configured hyperparameters (iterations, learning rate, densification)

Stage 8 — Export (PLY & SPLAT)

PM Lead Dev
Pipeline Stages ⑦ + ⑧ · Export

Exporting the Trained Model

The trained 3D Gaussian model is exported in two formats: PLY (standard 3D point cloud format, widely supported by 3D tools) and SPLAT (compact binary format optimized for real-time web rendering). Both files are written to the output directory with timestamp-based filenames.

Inputs
  • BackendOutput (trained model)
  • Output directory path
Outputs
  • model.ply (3D point cloud)
  • model.splat (web-optimized)
Output Formats

PLY — Industry-standard polygon file format. Compatible with MeshLab, Blender, CloudCompare, and other 3D tools. Contains position, color (SH coefficients), opacity, and covariance for each Gaussian.

SPLAT — Compact binary format designed for real-time rendering in web browsers. Significantly smaller file size with quantized attributes.

Potential Errors
  • Output directory write permission denied
  • Disk full during export
  • Model data corruption (serialization failure)

Section 12

Stage 9 — Post-Processing & Cleanup

PM Lead Dev
Stage 9 · Finalization

Moving Results & Cleaning Up

After the pipeline completes (success or failure), the input folder is moved to the appropriate destination. A background retention scheduler periodically purges old results.

  1. On Success — The input folder is moved from INPUT_PATH/scene/ to PROCESSED_PATH/scene/. Name collisions are handled by appending timestamps.
  2. On Failure — The input folder is moved to ERROR_PATH/scene/.
  3. Temp Cleanup — Temporary files in TEMP_PATH are cleaned via RAII (TempDir auto-deletes on drop).
  4. Retention Scheduler — A background task runs every hour, deleting folders in processed/ and error/ older than RETENTION_DAYS (default 30).

Section 13

Graceful Shutdown

Lead Dev Core Dev
Lifecycle · Shutdown

Signal Handling & Ordered Teardown

The system handles SIGTERM and SIGINT for clean container orchestration. On receiving a shutdown signal, it sets an atomic flag that both event loops check, allowing in-progress work to complete before teardown. A 5-minute timeout ensures the process always exits, even if a stage hangs.

  1. Signal Received — SIGTERM/SIGINT sets the atomic ShutdownFlag.
  2. Watcher Exits — The watcher loop checks the flag each iteration and stops scanning.
  3. Processor Drains — The processor completes its current job (if any), then exits.
  4. Cleanup Scheduler Aborted — The retention scheduler task is cancelled.
  5. Azure Unmount — If Blobfuse2 was mounted, containers are unmounted cleanly.
  6. Health Server Stopped — The HTTP health endpoint is shut down.
  7. Timeout Guard — If all of the above takes > 5 minutes, the process exits forcefully.
Potential Errors
  • Azure unmount failure during shutdown (logged as warning, non-fatal)
  • Shutdown timeout exceeded — process exits with code 1
