3D Gaussian Splatting doesn't start from nothing. Before a single Gaussian is trained, COLMAP must figure out where every camera was standing and build a rough 3D skeleton from feature matches. §01

What is Structure-from-Motion?

A photograph destroys depth — the 3D world is flattened onto a 2D sensor and all distance information is lost. But shoot the same scene from multiple angles and depth becomes recoverable. The trick is triangulation: if the same physical point is visible in two photographs, and you know where each camera was, you can trace a ray from each camera and find where those rays intersect in 3D.

Structure-from-Motion (SfM) automates this entirely — from a pile of unordered photos it simultaneously solves for every camera's position and reconstructs thousands of 3D landmark points. No GPS, no markers, no manual measurement. Just photographs and linear algebra.

Core concept — triangulation from two cameras

The orange cube corner appears in both photos · Each camera defines a ray into 3D space · Where rays cross = the 3D position · COLMAP does this for tens of thousands of points simultaneously

Why do you need so many photos? Each 3D point needs to appear in at least 2 images to be triangulated, but in practice 4–8 overlapping views makes it well-constrained and robust to noise. COLMAP recommends 60–80% overlap between consecutive shots.
§02

COLMAP — Six Stages from Photos to Points

Click any step to see what COLMAP is doing at that stage. Each stage feeds into the next.

1
Feature Extraction (SIFT)
COLMAP runs the SIFT algorithm on every photo. SIFT hunts for distinctive visual landmarks — corners, blobs, edges that are stable under changes in lighting, scale, and viewpoint. Each landmark (keypoint) gets a 128-number descriptor — a fingerprint of the local image patch. A single 12-megapixel photo typically yields 10,000–50,000 keypoints.
2
Feature Matching
Now COLMAP asks: "which keypoint in Photo A is the same physical point as which keypoint in Photo B?" It compares the 128-number descriptors — close descriptors = likely same point. This is O(N²) in the number of image pairs, so COLMAP uses approximate nearest-neighbour search (FLANN) to stay tractable. Only pairs with many mutual matches (>15) are kept.
3
Geometric Verification (RANSAC)
Raw descriptor matches are noisy — some are just wrong. COLMAP runs RANSAC (Random Sample Consensus): repeatedly sample 8 random match pairs, compute the fundamental matrix F they imply, count how many other matches agree with that geometry (inliers). The hypothesis with the most inliers wins. Outliers — matches that don't fit any consistent geometry — are thrown away.
4
Incremental Reconstruction
Start with the image pair that has the most verified matches. Recover their relative camera poses from the fundamental matrix, triangulate the shared points into 3D. Then greedily add images one by one: each new image must have enough already-triangulated 3D points visible in it to solve for its camera position using the PnP (Perspective-n-Point) algorithm. As cameras are added, more points can be triangulated.
5
Bundle Adjustment
All camera poses and 3D point positions accumulated so far have small errors that compound. Bundle adjustment runs a global optimisation: jointly tweak every camera pose and every 3D point position to minimise reprojection error — the pixel distance between where each 3D point projects to and where it was actually observed in each photo. Uses sparse Levenberg-Marquardt. Run every ~10 images added, and once at the very end.
6
Sparse Point Cloud Output
The finished product: calibrated camera poses (position + orientation + intrinsics) for every input photo, plus a sparse 3D point cloud (typically 50K–500K points). This is saved as a .ply file and a cameras.bin — the two inputs that 3DGS needs to initialise. One Gaussian will be seeded at each 3D point.

§03

Bundle Adjustment — The Critical Refinement

Bundle adjustment is the step that separates a rough reconstruction from a precise one. The name comes from the "bundle of rays" from each camera to each 3D point — the optimiser adjusts both ends of every ray simultaneously.

For each 3D point P and each camera C that observed it, bundle adjustment asks: if I project P through camera C using the current camera model, does it land where we actually saw it in the photo? The pixel-distance error is the reprojection error. Minimise the sum of all reprojection errors squared across all points and all cameras.

Reprojection error — what bundle adjustment minimises

Blue = observed position in photo · Orange = where 3D point currently projects to · Red line = reprojection error · Bundle adjustment drives all red lines to near-zero by moving cameras AND 3D points simultaneously

Before bundle adjustment

Each image registered independently accumulates small errors. A point triangulated from cameras 1 and 2 lands in a slightly wrong position when projected into camera 5. These errors compound across the reconstruction.

After bundle adjustment

Global optimisation finds the joint assignment of camera poses and 3D positions that is most consistent with all observations at once. Typical indoor scenes achieve sub-pixel reprojection error after convergence.

§04

The Handoff — Sparse Cloud Becomes Gaussians

COLMAP hands 3DGS two things: a .ply file of 3D points and a file of camera poses. 3DGS reads these and seeds one Gaussian at each 3D point. The initial values are set to reasonable defaults — not optimised, just a sensible starting point so gradients flow in meaningful directions from iteration one.

Why sparse initialisation works

The sparse cloud doesn't need to be accurate or dense — just topologically correct. As long as Gaussians start near the right geometry, gradients will push them toward the right positions.

3DGS densifies during training: starting from ~100K sparse points it grows to 3–6 million Gaussians by iteration 15,000. The COLMAP seed saves roughly 5–10× training time versus random initialisation.

One practical implication: regions COLMAP misses become regions 3DGS struggles with. Featureless surfaces (white walls, sky, water) give COLMAP no keypoints → no seed Gaussians → 3DGS must densify from nothing.

§05

How It Learns — The Training Loop

Every training step follows the same four-beat rhythm, repeated 30,000 times. The loop is simple in structure but involves millions of parameters being nudged simultaneously. Watch each stage light up as execution passes through it.

Live animation — one training iteration (×30,000)

Picking a training camera from the COLMAP-registered dataset…

Step 1 — Pick a Training Camera

One photo is sampled at random. Its COLMAP-recovered pose tells us exactly where the virtual camera is: position in 3D, viewing direction, focal length. This is the "ground truth" viewpoint we'll render from.

Step 2 — Rasterise All Gaussians

Every Gaussian in the scene is projected to 2D screen coordinates. They're then depth-sorted (back to front) and alpha-composited tile by tile into a rendered image. This entire process is differentiable — gradients can flow back through it.

Step 3 — Compute Loss

The rendered image is compared to the real photo pixel-by-pixel:

L = 0.8 × L₁(render, photo) + 0.2 × (1 − SSIM)

L₁ catches colour errors. SSIM (Structural Similarity) catches structural and edge errors that L₁ misses. The combined loss gives better-looking results than either alone.

Step 4 — Backprop + Adam Update

Gradients flow backward through the rasteriser to every Gaussian parameter — all 59 numbers per blob (position, scale, rotation, opacity, spherical harmonic coefficients). The Adam optimiser updates each parameter with a per-parameter adaptive learning rate, nudging the scene toward lower loss.

The densification interrupt. Every 100 steps (between iterations 0–15,000), 3DGS pauses the main loop to inspect position gradients. Gaussians that accumulated large gradients live in under-reconstructed regions and need more blobs. They're either cloned (if small) or split (if large). Every 3,000 steps, all opacities are reset toward zero, forcing each Gaussian to re-earn its existence.

Loss & Convergence

The loss curve tells the story of training. Three distinct phases are visible in every 3DGS run. Hover the chart to inspect exact values at any iteration.

Training loss over 30,000 iterations — hover to inspect

PSNR typically reaches 28–32 dB for indoor scenes after full training

Phase 1 · 0–3,000 iters · Rapid Descent

Gaussians move from COLMAP seed positions toward correct geometry. Huge obvious colour and position errors are corrected. Loss drops by ~80% of its total fall in this phase alone. The scene looks blurry but structurally correct.

Phase 2 · 3K–15K iters · Densification

Clone/split events add new Gaussians every 100 steps. Each burst briefly increases loss (more competing blobs), then recovers to a lower baseline. Scene count grows from ~200K to ~2–4M Gaussians. Fine details emerge.

Phase 3 · 15K–30K iters · Fine Refinement

Densification stops. The Gaussian count is fixed. Remaining improvement comes from tuning spherical harmonic colour coefficients (view-dependent colour), fine-tuning opacity and shape. PSNR plateaus. Stopping here versus running longer is a quality/time tradeoff — most scenes don't improve much past 30K iters.

§07

Densification — Clone, Split, Prune

The most novel part of 3DGS is its adaptive density control. Rather than fixing the number of Gaussians upfront, the system grows its own geometry wherever the current representation is failing.

The three densification decisions

Trigger: position gradient magnitude exceeds threshold (default 0.0002) · Decision branches on Gaussian size relative to scene scale

Clone — small Gaussian, high gradient

A small Gaussian is in an under-reconstructed area (high gradient because rendered colour doesn't match the photo). The fix is more coverage. Solution: duplicate the Gaussian and offset the clone slightly in the gradient direction. Two small blobs now cover what one couldn't.

Split — large Gaussian, high gradient

A large Gaussian spans a region with fine detail it can't capture (still high gradient). The fix is finer resolution. Solution: replace the large Gaussian with two smaller ones (scale ÷ 1.6) placed along its principal axis. The two children can now capture the variation the parent couldn't.

Prune — opacity near zero

Gaussians that fell to near-zero opacity during training are invisible and contributing nothing. They're simply deleted. This is triggered every 100 steps by checking sigmoid(logit_opacity) < 0.005. The opacity reset every 3,000 steps forces this to happen regularly — any Gaussian that isn't genuinely useful will drain back toward zero and get pruned.
